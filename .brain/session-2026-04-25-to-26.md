## Lessons

### L-001: `useEffect([])` + async state writes silently fail with "run once when ready" patterns
- **Surface:** Wizard step state was reset on remount because the load effect ran before profile fetch resolved.
- **Bug:** Empty dep array meant the effect didn't re-fire when the awaited data eventually arrived; the post-await `setState` landed on a stale closure.
- **Fix:** Add a `ready` boolean to deps so the effect re-runs once the data lands; or move the await inside an effect that itself depends on the loaded slice.
- **Transferable rule:** Async work inside `useEffect([])` only safely fires-and-forgets. If the effect needs to call `setState` based on awaited data, list the data's readiness in deps or you've built a silent race.
- **Tags:** [react, useeffect, async]

### L-002: Codex argv overflows on large diffs — pipe via stdin
- **Surface:** Pre-commit hook silently truncated a 847-insertion diff being passed as a CLI argument.
- **Bug:** OS argv length cap (varies by platform, ~32K-128K) cut the diff mid-line; codex received valid-looking but partial input.
- **Fix:** `printf '%s' "$DIFF" | codex …` instead of `codex "… $DIFF"`.
- **Transferable rule:** Any time you're shelling out with payload-size-unbounded input, use stdin. Argv is for flags.
- **Tags:** [shell, ipc, payload-size]

### L-003: `claude -p` has a hard timeout that subagent dispatch doesn't
- **Surface:** Hook reviewer step hung past timeout on the Phase 1 diff.
- **Bug:** `-p` mode is intended for short prompts; long input hits the wall.
- **Fix:** Dispatch a subagent (`Agent` tool) with the diff in the prompt body.
- **Transferable rule:** When integrating Claude into pipelines, choose the entry point by input size: `-p` for tiny, subagent for non-trivial.
- **Tags:** [claude, timeout, integration]

### L-004: Severity-flag absence ≠ clean review — default to BLOCK on integrity
- **Surface:** First hook iteration parsed reviewer output for `severity: critical|high|...` and treated absence as PROCEED.
- **Bug:** A reviewer that crashed or returned empty produced output identical to a clean review.
- **Fix:** Require an explicit `status: ok` token; absence blocks. Plus a separate "both reviewers skipped" guard.
- **Transferable rule:** Fail-closed gates require an explicit success signal, never inferred-from-absence. The empty-and-clean output ambiguity is the standard injection point.
- **Tags:** [fail-closed, integrity, hooks]

### L-005: Regex for "no findings" must tolerate markdown bold and reviewer-specific phrasing
- **Surface:** Hook integrity check missed a clean review because Codex wrote `**No findings.**` and the regex only matched `No findings.`.
- **Bug:** `\bNo findings\.` doesn't match `**No findings.**` — the asterisks are word-boundary breakers.
- **Fix:** `\*?\*?No findings\.\*?\*?` (or strip markdown before matching).
- **Transferable rule:** When parsing free-form text from multiple producers, normalize first (strip formatting), then pattern-match on the normalized form.
- **Tags:** [regex, parsing, normalization]

### L-006: `String.replace` with user input is an injection vector via `$&` `$1` `$$`
- **Surface:** Token substitution in `player-tokens.ts` initially used string-form replacement.
- **Bug:** A `display_name` of `$&` would echo the matched token (`{playerName}`) back into the rendered text. `$1` etc. similarly hostile.
- **Fix:** Function-form `(_, …) => profile.display_name`. The replacement function receives the user input as a closure and is opaque to `$` sequences.
- **Transferable rule:** Any `String.replace` whose replacement contains user-controlled data must use the function form. The string form interprets `$` patterns.
- **Tags:** [security, javascript, user-input]

### L-007: Exam date timezone off-by-one — `toISOString` vs `format(d, 'yyyy-MM-dd')`
- **Surface:** Date picked in the wizard saved one day earlier in the DB.
- **Bug:** `new Date(yyyy, mm, dd).toISOString()` converts to UTC; in negative-offset timezones the date rolls back.
- **Fix:** `format(date, 'yyyy-MM-dd')` from `date-fns` — preserves local-day semantics.
- **Transferable rule:** Date-only fields must be serialized with a date-only formatter, never via `toISOString` (which is a datetime).
- **Tags:** [datetime, timezone, serialization]

## Prompts that worked

### P-001: "Decisions before execution" spec hand-off
- **Used in:** Spec phase (Claude.ai) → execute phase (Claude Code) for Phase 1
- **Pattern:** Spec doc enumerates every decision (schema columns, enum values, fallback chain) so the build agent never re-decides during execution.
- **Why it worked:** Build agents drift when forced to make design calls under context pressure. Pre-deciding moves judgment to the slow, deliberate channel where it belongs.
- **Reusable form:**
  ```
  Before you write any code, list every design decision the build will hit
  (schema, enum values, fallback behavior, error UX). For each: state the chosen
  option and a one-sentence why. The build phase implements; it does not decide.
  ```
- **Output quality:** Phase 1 shipped with zero mid-build redesigns. The one re-decision (`nonbinary` add) was explicit, not drift.
- **Tags:** [workflow, spec, hand-off]

### P-002: Parallel manual review of single commit using subagent for the Claude side
- **Used in:** Manual review of 9cad25c (pre-amend) when the pre-commit hook had failed silently
- **Pattern:** Dispatch a subagent to do the Claude review while running the Codex review inline; merge findings.
- **Why it worked:** Subagents have isolated context and don't pollute the main thread with diff noise. Inline Codex is fast enough not to need its own subagent.
- **Reusable form:**
  ```
  Subagent prompt: "Review the diff below as a senior reviewer. Output findings
  in severity order (Critical/High/Med/Low). End with 'status: ok' if no
  Critical or High. Diff: <full diff>."
  ```
- **Output quality:** Caught the real Critical (`ONBOARDING_PATH` mismatch) and produced two false positives that were quickly verified away.
- **Tags:** [review, subagent, parallel]

### P-003: Operating-rules block prepended to multi-step build prompts
- **Used in:** Phase 1 wizard build
- **Pattern:** A standing block at the top of the build prompt: "one file at a time, surface decisions, no autopilot."
- **Why it worked:** Counters the build agent's default of barrelling through multi-file changes silently. Forces a checkpoint cadence.
- **Reusable form:**
  ```
  Operating rules for this session:
  - Edit one file at a time. Show the diff before moving on.
  - Surface any decision not pre-specified in the spec. Do not infer.
  - No autopilot: if you hit ambiguity, stop and ask.
  ```
- **Output quality:** Two ambiguities surfaced cleanly (reflexive pronoun, exam_date column) instead of silent invention.
- **Tags:** [meta-prompt, control, cadence]

### P-004: Confidence-flag pattern (VERIFIED / LIKELY / UNVERIFIED) for review claims
- **Used in:** Reviewing Codex findings before deciding to amend
- **Pattern:** Tag every claim about the diff as VERIFIED (re-checked the file), LIKELY (consistent with diff but not re-checked), or UNVERIFIED (inference).
- **Why it worked:** Forces the reviewer (and me) to separate "I read it" from "I'm guessing." Two of three Codex Criticals turned out UNVERIFIED on inspection.
- **Reusable form:**
  ```
  For each finding, append a confidence tag:
  - VERIFIED: I re-read the cited file/line.
  - LIKELY: consistent with the diff, not re-checked.
  - UNVERIFIED: inference from naming or pattern matching.
  ```
- **Output quality:** Stopped me from amending for two false positives.
- **Tags:** [verification, confidence, review]
