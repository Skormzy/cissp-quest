## Critical Issues Blocking Stripe (from Codex Adversarial Review, 2026-04-19)

These 12 findings live in `REVIEW_FINDINGS_2026-04-19.md`. Must fix before Stripe ships.

### Critical (blocking ship)
1. **Missing DB columns** ÔÇö `stripe_customer_id`, `stripe_subscription_id`, `subscription_expires_at` don't exist in migration SQL. Users who pay get nothing.
2. **Subscription tier client-writable** ÔÇö RLS policy has no column restriction. Any user can self-upgrade via browser console.
3. **Webhook signature verification optional** ÔÇö Falls back to `JSON.parse(body)` if `STRIPE_WEBHOOK_SECRET` missing. Anyone can forge a payment event.

### High (urgent)
4. Webhook DB failures silently return 200. Failed upgrades are lost permanently.
5. Paid-feature gates are client-only ÔÇö Ghost POV, Evidence Constellation, Practice Exam have no server-side tier check. Direct URL bypasses them.
6. Community/notification tables (`chatroom_messages`, `weekly_scores`, `notification_queue`) referenced in code but not in migration SQL. These pages will 500 on deploy.
7. Annual billing UI does not reach the server (param name mismatch: UI sends `annual`, server reads `priceId/mode/tier`).

### Medium
8. `success_url` trusts request Origin header (open redirect risk).
9. Subscription lifecycle doesn't check status field (past-due subscriptions keep access).
10. CAT domain weights configured but not enforced in `selectNext()`.
11. SPRT helper pass/fail logic is inverted in `irt-engine.ts:267`.
12. `seed-questions.ts` is destructive and silently swallows errors.

**Priority fix order (from review):** 1 Ôćĺ 2 Ôćĺ 3 Ôćĺ 5 Ôćĺ 4 Ôćĺ 7 Ôćĺ 8 Ôćĺ 9 Ôćĺ 11 Ôćĺ 10 + 12

## The Strategic Fork: Fix First vs. Rebuild First

This is the key decision for the next session. Two paths:

### Path A: Fix the Broken Stripe First, Then Rebuild
- Fix the 12 adversarial review findings (1-2 sessions)
- Stripe goes live with current UI
- Then execute full rebuild on top of a working monetization layer
- Risk: Doing the rebuild after Stripe means rebuilding around payment flows that may change

### Path B: Execute the Full Rebuild (Stripe integrated from the start)
- Install shadcn/ui (critical first step, currently missing)
- Rebuild UI from scratch with new narrative and correct Stripe integration baked in
- The 12 findings get addressed as part of the rebuild ÔÇö not as a separate patch
- Risk: Longer to first revenue, but no double work

**Current lean (based on rebuild spec design intent):** Path B. The rebuild is the product. Patching the old UI and then rebuilding it wastes cycles. The rebuild spec explicitly accounts for Stripe being done correctly from scratch.

## Key Design Principles (Non-Negotiable)

- **shadcn/ui is the foundation.** Every structural UI element (forms, modals, inputs, cards, buttons, tables, toasts) uses shadcn. Custom visual layers (anime aesthetic, dark theme, illustrations) go ON TOP of shadcn, not instead of it.
- **Teach first, test second.** Wrong answers always link back to a Library topic with a Memory Hack.
- **Think Like a Manager.** CISSP candidates must understand that the exam tests governance and process, not unilateral technical action. This framing is embedded throughout the narrative.
- **Original content only.** All questions, explanations, and Library entries are original prose informed by ISC2 outline and reference textbooks ÔÇö never copy-pasted.
- **One shared narrative.** 8 Acts, one story, one character arc. Not 8 isolated mini-stories.
- **Monetization structure:** Free tier (Act 1 + Domain 1 quiz), one-time full unlock, Pro subscription with spaced repetition and CAT, Enterprise per-seat.
